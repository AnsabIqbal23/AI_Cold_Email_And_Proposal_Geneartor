# AI Cold Email / Proposal Generator — Project Spec & Conventions

## What This App Does

A web app where a user enters a client's business information and receives a tailored cold sales email or structured proposal generated by an LLM. Target users are freelancers and small sales teams.

---

## Stack (do not deviate without telling the user)

- **Framework**: Next.js (latest), App Router, TypeScript
- **Styling**: Tailwind CSS
- **LLM default**: Google Gemini Flash
- **LLM fallback**: Groq
- **Deployment**: Vercel free tier
- **Database**: None in v1 — all state lives client-side per session
- **Budget**: Zero — free tiers only

---

## LLM Architecture

- All LLM calls happen **server-side only**. API keys must never reach the browser.
- The LLM layer must be **fully provider-agnostic**: the rest of the app never imports a specific provider SDK directly. Provider selection is encapsulated in `lib/llm/`.
- Default provider: **Google Gemini Flash**. Fallback: **Groq**.

---

## Directory Conventions

```
app/              Next.js App Router pages and API routes
components/       UI only — no business logic, no direct API calls
lib/
  llm/            Provider-agnostic LLM layer (the only place that knows about Gemini/Groq)
  prompts/        Prompt construction and template logic
  types.ts        All shared TypeScript types (single source of truth)
```

- `lib/llm/` — provider selection, model calls, error handling, fallback logic
- `lib/prompts/` — builds and formats prompts from user input
- `components/` — receives data and callbacks as props, calls nothing directly
- Business logic lives in `lib/`, UI lives in `components/`, these never cross

---

## Code Quality Rules (apply to every file, every phase)

1. **One responsibility per file** — if a file does two things, split it.
2. **Files stay small** — if a file is getting long, it is a signal to extract.
3. **Descriptive names** — no clever abbreviations. `generateEmailPrompt` not `genEP`.
4. **No logic in components** — components call functions from `lib/`, never implement logic themselves.
5. **Single type source** — all shared types go in `lib/types.ts`. Do not redeclare types elsewhere.
6. **No duplicated logic** — factor shared code out before writing it a second time.
7. **Comments explain why, not what** — only add a comment if the reason is non-obvious. Never narrate obvious code.
8. **Clarity over cleverness** — the most readable solution wins, always.

---

## Output Conventions for Generated Copy

- Emails must read like a real human wrote them.
- **Never use em-dashes** (`—`).
- **Never use emojis**.
- Avoid stiff AI-sounding openers like "I hope this email finds you well".
- Avoid overly formal or corporate phrasing.

---

## Environment Variables

All keys and config values go in environment variables. Document every variable in `.env.example` as it is introduced. Never hardcode secrets.

---

## What Each Phase Must Respect

Every phase of the build must follow all rules above — code quality, directory structure, type conventions, and output conventions are not phase-specific, they apply from the first line of code written.
